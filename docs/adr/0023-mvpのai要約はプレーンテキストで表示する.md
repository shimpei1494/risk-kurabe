# ADR-0023: MVPのAI要約はプレーンテキストで表示する

- 状態: 採用
- 決定日: 2026-07-17

## コンテキスト

AI要約は公開データによる地点調査と事実比較を読みやすくする補助機能である。構造化JSONを複数のUI部品へ変換する実装は表示を豊かにできる一方、2週間のハッカソン版ではスキーマ設計、出力修復および複数表示状態の検証が追加で必要になる。

将来はOpenUIを用い、AIが許可済みUIコンポーネントを構成するリッチな表示を検討している。OpenUIはLLMにコンポーネントライブラリを含むシステムプロンプトを渡し、OpenUI Langの応答をReact Rendererで描画する独立したGenerative UIフレームワークである。

## 決定

- MVPのAI応答は1つのプレーンテキストとして受け取り、表示する。
- MVPではストリーミングを使用せず、応答全文の受信と検証が完了してから表示する。
- 応答では短い段落と改行を許可するが、HTML、任意のReact要素、OpenUI LangおよびJSON UI定義を生成させない。
- 画面ではテキストをエスケープしたまま表示し、`dangerouslySetInnerHTML`を使用しない。
- 応答本文は最大800文字とし、プロンプトとアプリ側検証の両方で制限する。
- WorkerからOpenAIへの生成処理は20秒でタイムアウトさせる。
- 生成中は進行中表示を出し、完了後に本文をまとめて表示する。
- タイムアウトまたは生成失敗時はユーザー操作による再試行を提示し、自動再生成しない。
- AI入力には、地点判定、浸水深区分、データ状態、境界警告および根拠識別子からなる版付きの正規化済み事実スキーマを使用する。
- AI出力はUI状態や地点判定の入力として再利用しない。
- 空応答、上限を超える応答または禁止された推奨表現を検出した場合は本文を表示せず、生成失敗として再試行操作を提示する。
- OpenUIの依存関係、Renderer、コンポーネント定義および生成用プロンプトはMVPへ含めない。

## 将来のOpenUI対応

- OpenUI導入時も、公開データによる地点判定と正規化済み事実スキーマを入力の基準にする。
- AIが利用できるコンポーネントを、要約、地点差分、注意事項および根拠リンクなどの読み取り専用部品へ制限する。
- Mantineで実装したアプリ固有コンポーネントをOpenUIへ登録できるか、TanStack StartとCloudflare上で技術検証する。
- モデルが任意コード、任意URLまたは地点判定を生成・実行できる設計にはしない。
- 導入時点のOpenUI Lang、React対応、ライセンス、Cloudflare互換性およびバージョン安定性を改めて確認する。

## 結果

- MVPのAI表示とエラー処理が小さくなり、公開データの表示開発を優先できる。
- プレーンテキストの構成はモデル出力に依存するため、リッチな視覚的一貫性は提供しない。
- ストリーミングによる逐次表示は行わないが、短い要約に限定することで待ち時間と実装量のバランスを取る。
- 正規化済み事実スキーマをAI表示から分離することで、将来OpenUIへ移行してもGIS判定を作り直さずに済む。
- OpenUI移行時には応答形式、ストリーミング、キャッシュキーおよび出力検証版を変更する必要がある。
