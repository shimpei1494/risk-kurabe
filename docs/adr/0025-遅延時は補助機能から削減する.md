# ADR-0025: 遅延時は補助機能から削減する

- 状態: 採用
- 決定日: 2026-07-17

## コンテキスト

ハッカソン版は1人がAIを活用して2週間で開発する。公開GISデータの加工、ブラウザ上の空間判定、住所検索、比較UIおよびCloudflareへの配置を同時に進めるため、予期しないデータ品質や互換性の問題が起こり得る。

期限直前に場当たり的に機能を削ると、データ状態や根拠表示など、アプリの信頼性に必要な部分まで欠ける可能性がある。そのため、削減対象と維持対象を実装前に決める。

## 決定

進捗が計画より遅れた場合は、次の順にMVPから削減する。

1. 二択フィードバックと詳細な独自分析イベント
2. AI要約
3. 最近使った地点とURLフラグメントによる簡易共有
4. 広域PMTilesレイヤー表示。地点調査に必要なFlatGeobuf判定と、可能ならピン周辺の根拠ポリゴン表示は維持する

次の機能と品質要件は最後まで削減しない。

- 1地点の調査と、最大3地点への段階的な比較
- 関東のA31a想定最大規模洪水浸水深区分
- 関東で公開・収録できるA53頻度別洪水浸水深区分
- 東京都の建物倒壊危険度と火災危険度
- `区域外`、`未公開`、`対象外`および`判定不能`の区別
- 重複時の最大浸水深区分と全根拠の保持
- 指標ごとの半径25m境界警告
- 出典、データ基準時点および注意事項
- 外部API障害時にも公開データによる結果を利用できること

## 運用ルール

- 機能削減は、依存する未完成コードを残さず、画面、ルート、設定および分析イベントから一体で外す。
- 削減判断は日次の完了条件を基準に行い、期限前日に初めて判断しない。
- P0機能のデータ正確性を下げてP1・P2機能を残さない。
- 削減した機能は、デモ資料や画面上で実装済みのように見せない。

## 結果

- 期限内に、比較の根拠と限界を説明できる中核体験を優先できる。
- AIや共有を削っても、公開データ活用アプリとして地点調査と比較を成立させられる。
- 広域地図を削る場合でも地点判定は維持するため、R2のFlatGeobuf成果物をPMTilesより先に完成させる必要がある。
