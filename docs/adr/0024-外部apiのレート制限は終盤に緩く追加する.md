# ADR-0024: 外部APIのレート制限は終盤に緩く追加する

- 状態: 採用
- 決定日: 2026-07-17

## コンテキスト

ログイン不要の公開アプリからYahoo住所検索とOpenAI要約を利用するため、APIキーはWorker内に隠しても公開エンドポイントの連打や自動アクセスは起こり得る。一方、ハッカソン版では強い利用制限によって試用しにくくしたり、主要機能の実装より先に高度な不正利用対策へ時間を使ったりしない。

Cloudflare WorkersのRate Limiting Bindingは短時間のバースト制御に利用できるが、拠点単位で結果整合性が緩く、厳密な使用量集計や課金制御を目的とした仕組みではない。

## 決定

- APIキーはCloudflare WorkerのSecretとして保持し、ブラウザへ配布しない。
- Yahoo住所検索とOpenAI要約のWorker経路に、Rate Limiting Bindingを使った緩いバースト制限を追加する。
- Bindingの期間は10秒とする。
- 初期値は、OpenAI要約を同一キーあたり10秒に5回、Yahoo住所検索を同一キーあたり10秒に20回とする。
- 匿名MVPでは安定したユーザーIDがないため、キーは`経路種別 + 接続元IP`を使用する。
- IPはRate Limiting Bindingへ渡すためだけにリクエスト中に利用し、アプリのログ、Analytics Engine、localStorageまたはデータベースへ保存しない。
- 制限超過時はHTTP 429と一般的な再試行案内を返し、待ち時間を厳密に保証しない。
- Turnstile、KV、D1または独自の利用量カウンターはハッカソン版へ導入しない。
- レート制限の実装は、住所検索とAI要約が正常に動作し、主要な地点調査・比較機能が完成した後に行う。

## 実装順

1. 入力スキーマ、本文サイズ、許可メソッドおよびOriginを検証する。
2. 外部API呼び出しを実装し、正常系とエラー変換を確認する。
3. アプリの主要導線を完成させる。
4. Rate Limiting Bindingを追加し、429表示と匿名計測を確認する。

## 結果

- 通常の試用を妨げにくい範囲で、単純な連打と短時間の大量アクセスを抑制できる。
- IP共有環境では複数ユーザーが同じ制限キーになる可能性があるが、緩い上限によって影響を抑える。
- Bindingは厳密な利用量制御ではないため、OpenAI側の予算・使用量設定と利用状況の監視は別途必要になる。
- 実運用へ移行する場合は、認証、Turnstile、より長い期間の制御または全体予算保護を改めて設計する。
