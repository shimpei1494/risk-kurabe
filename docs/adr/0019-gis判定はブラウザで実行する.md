# ADR-0019: GIS判定はブラウザで実行する

- 状態: 採用
- 決定日: 2026-07-17

## コンテキスト

ハッカソン版の公開GISデータはR2上のFlatGeobufとPMTilesとして配信する。地点判定をWorkerで行う場合、地点ごとのWorker処理、座標送信および応答待ちが発生する。一方、データは公開情報であり、サーバー側で形状や判定規則を秘匿する要件はない。

2週間の一人開発では、ブラウザ用とWorker用に二重のGIS実装を持たず、責務を明確に分ける必要がある。

## 決定

- 地点判定、重複ポリゴンの解決、A31aとA53の関連付けおよび半径25mの境界検査はブラウザで実行する。
- ブラウザはデータマニフェストとカバレッジ情報を読み、必要な都県・データ種別のFlatGeobufだけをRange取得する。
- 地図用PMTilesもR2からブラウザへ直接配信し、MapLibreで表示する。
- Cloudflare WorkerはYahoo住所検索、OpenAI要約および匿名利用計測など、秘密鍵またはサーバー処理が必要な機能だけを担当する。
- 地点判定のために住所または座標をWorkerへ送信しない。
- ブラウザとWorkerは同じ判定コードを重複して持たず、判定結果の正規化スキーマだけをAI要約処理との境界として共有する。

## 配信要件

- FlatGeobufとPMTilesは版付きの不変URLで公開する。
- R2配信ではHTTP Range要求を許可する。
- 別オリジンで配信する場合は、アプリのオリジンだけを許可するCORSを設定する。
- CORSでは少なくとも`Range`要求ヘッダーを許可し、`Accept-Ranges`、`Content-Range`、`Content-Length`および`ETag`をブラウザから参照可能にする。
- 都県とデータ種別を基本単位に成果物を分割し、地点と無関係な地域の取得を避ける。
- 取得失敗、非対応ブラウザまたは破損した成果物は`判定不能`として扱い、区域外とは表示しない。

## 性能検証

- 実データによる技術検証を実装初期に行う。
- 1地点の初回判定が一般的なスマートフォン相当環境で概ね1秒を超える場合は、まずFlatGeobufの分割単位、属性量および形状簡略化を調整する。
- 同じタブ内の再判定はブラウザキャッシュとsessionStorageを利用する。
- 性能問題を理由に直ちにWorker判定やD1へ切り替えず、計測結果を残して別ADRで判断する。

## 結果

- WorkerのGIS処理負荷と地点ごとのAPIコストを抑えられる。
- 公開GISに対する地点判定では、ユーザーの住所・座標がアプリのWorkerへ送信されない。
- 端末性能とネットワーク速度の影響を受けるため、ローディング、失敗状態および再試行UIが必要になる。
- ブラウザへ判定規則が配布されるが、公開データを決定的に解釈する処理であり問題としない。
